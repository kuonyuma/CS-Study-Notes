# 1. 什么是阻塞队列


在以前，购物网站的发展其实还不够健全，服务器的每日访问量也不是特别大。
但是，每当双十一到来，购物网站的服务器就会面临极大的压力，这些压力由客户端直接发送给服务器，最终导致服务器崩溃。
于是，后来就引入了一个缓解服务器压力的机制——阻塞队列。
就拿卖包子来说。我开了一家包子店，店里做包子的就我一个人。
在平常，来的客人没有那么多，我的出货量刚好能满足需求，可以悠闲地做着包子。突然有一天，店里被一大波人围得水泄不通。
他们都抢着伸出手找我要包子。我哪经历过这样的大场面，直接汗流浃背了。
这种环境下还能做包子吗？肯定不能了，因为店里突然来了一大群人把店挤满了，我都没地方做了。
为了应对这种情况，我想了一个办法：在店门口放了一个篮子，专门用来放包子。这个篮子最多能放100个。
我只需把包子往篮子里放，客人只管从篮子里取包子。
在空闲的时候，我只管做，做完一个就往篮子里放；当篮子放满的时候，我就停下来等待。
当来了一个客人取走一个包子，我发现篮子没满，于是我又开始做；做满了，我又停下。
在生意火爆的时候，客人们纷纷从篮子里取包子，由于我做的速度没有他们取得快，当客人面临着篮子空空如也的情况时，他们就会等在篮子面前。一旦我往里放了一个包子，顾客看见就立即取走，然后下一个人继续等待。
自从加了这个篮子后，我只管面对这个篮子，专心做包子就行了。
我根本不用面对人流量的压力，可以安心地做着包子，篮子替我解决了流量暴增带来的压力。
这就是阻塞队列。

# 2. 阻塞队列的简单实现
我们不着急去看复杂的源码，
我们看看一个普通的队列是如何升级变成阻塞队列的。
```java

class MyQueue{
    //自己实现一个基本的整型队列
    private int[] array;
    private int left,right = 0;
    private int capacity = 100;
    //实际存储了多少个数
    private int size;
    public MyQueue(){
        array = new int[capacity];
    }

    public boolean put(int val) throws InterruptedException {
        synchronized(this){
            while(size == array.length){// 数组满了阻塞等待
                this.wait();
            }
            array[right] = val;
            size++;
            this.notifyAll();
            // 为了提高空间利用率，我们要重复使用前面的位置
            right = (right + 1) % capacity;
            return true;

        }
    }

    public Integer pop() throws InterruptedException {
        synchronized(this){
            while(size == 0){
                System.out.println("开始阻塞");
                this.wait();
                System.out.println("阻塞结束");
            }
            int buf = array[left];
            left = (left + 1) % capacity;
            size--;
            this.notifyAll();
            return buf;
        }

    }
    public Integer peek(){
        if(size == 0){
            return null;
        }
        return array[left];
    }
}

```

字段很简单。
1. ``private int[] array;`` 这是一个基于数组实现的队列
2.  ``private int left,right = 0;``为了重复利用空余的位置
3.  ``private int capacity = 100;``确定队列的大小，这里为了简单直接设置为100
4. ``private int size;``队列的实际元素个数。

我们来看看``put``与``pop``这两个方法。

这里与普通的队列最大的不同是有一个``synchronized``和``notifyAll，wait``

我们先来说这个wait，
```java
while(size == array.length){// 数组满了阻塞等待
                this.wait();
            }
```

这里的含义是如果队列满了，我就停下来，直到队列出现空位，再补上。
也就是不往这个队列中添加数据了嘛，不就是阻塞了吗？
我们知道wait需要被一个唤醒，被谁唤醒呢？
就是pop中的notifyAll.
```java
 	size--;
 	this.notifyAll();
	return buf;
```
当pop触发时元素个数减少，那么我们就在size后面使用notifyAll，唤醒put中的wait.
让他知道队列空了，可以继续添加数据了。


pop中的wait与上述类似，逻辑相反。

为什么需要锁呢？

wait的使用必须和synchronized搭配。
wait是主动释放手里的锁，在释放的时候你肯定得先获得一把锁吧？
而且添加元素并不是原子性的，
如果不加上锁，会引发线程安全问题。

细心来看，我其实没有这样写``if(size == 0)``
这是为什么呢？
如果这里有两个线程A，B同时开始取出数据，
此时A线程运行，经过if判断发现里面没有包子，于是进行阻塞等待，
此时B线程运行经过if判断发现没有包子于是阻塞等待，
此时生产包子的线程C登场，使用了一个notifyAll,唤醒所有线程 A 和 B，
假设这里A先拿到锁，此时拿走一个包子，此时包子数量==0，
然后 B 也拿到了这把锁，由于之前在 if 块中已经判断过一次了，此时不会再次检查 size，而是直接去拿包子，结果报错了....

如果加上一个while，线程AB会重复判断一下当前的包子个数，线程B会发现包子==0而退出循环。

这里还要注意为什么使用notifyAll,而不是单纯的notify

假设队列的容量是 100（capacity = 100）。
1. 队列已经满了（size = 100）。
2. 此时有 2 个生产者线程（A 和 B）尝试往队列里 put() 包子，它们都发现队列满了，于是双双进入 wait() 状态。
3. 同时，有 1 个消费者线程（C）尝试 pop() 拿包子。消费者 C 成功拿走了一个包子，队列空出一个位置（size = 99），然后 C 执行了 this.notify()。
4. notify() 随机唤醒了一个等待的线程，比如唤醒了生产者 A。
5. 生产者 A 发现队列有空位了，它往里面放了一个包子。此时队列又满了（size = 100）。然后 A 也执行了 this.notify()，试图告诉别人：“我放好包子了，你们可以来拿或者来放了”。
6. 灾难发生：A 执行的 notify() 究竟唤醒了谁呢？
   - 如果运气好，它唤醒了消费者 C，消费者 C 拿走包子，一切正常。
   - 但是！如果运气不好，A 的 notify() 把剩下的那个生产者 B 给唤醒了！
7. 生产者 B 醒来，检查 while (size == array.length)，发现队列竟然还是满的（因为刚才 A 已经把它填满了），于是 B 再次进入 wait() 状态。
8. 而此时唯一的消费者 C 还在等待（它没被唤醒过，或者已经处理完进入等待了）。
   - **结果**：生产者 A 放完了，生产者 B 醒了又睡了，消费者 C 没人唤醒。所有线程都进入了 wait 状态，程序锁死（假死）。

这就是为什么要用 `notifyAll()`。它会唤醒**所有**等待的线程。即使唤醒了不符合条件的生产者，它们检查 `while` 后会继续睡去，但**至少能保证消费者也一定会被唤醒**，从而维持程序的运转。

# 总结
本文主要介绍了阻塞队列的概念及其实现原理。
1. **阻塞机制**：通过 `wait()` 和 `notifyAll()` 实现了生产消费者模型。
2. **锁的作用**：保证了原子性并配合 `wait` 使用。
3. **while 检查**：防止虚假唤醒导致的逻辑错误。
4. **notifyAll vs notify**：防止信号丢失导致的程序假死。
