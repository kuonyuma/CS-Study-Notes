# 1.死锁
## 1.1 什么是死锁？
大家都吃过饺子吧。在我们这里，吃饺子必须得蘸酱油（或者醋？）。
现在你和你的女朋友正围着桌子。
桌子上有一盘饺子，一叠醋。
你拿走了那盘饺子，你的女朋友拿走拿一叠醋。
由于饺子和醋必须配合起来才能吃。
你不愿意放下手里的饺子，你的女朋友也不愿放下手里的醋。
场面就僵住了。
你虽然有饺子但是没有醋。
你女朋友虽然有醋但是没有饺子。
谁也不肯让谁。
构成了一个死结。
**这种情况就是死锁**。
## 1.2 一个简单的思索案例
我们先看看这个代码。
```java

    public static void main(String[] args) throws InterruptedException {
       // 写一个死锁案例
        Object locker1 = new Object();
        Object locker2 = new Object();

        Thread t1 = new Thread(()->{
            synchronized(locker1){
                System.out.println("三叶_拿起第一把锁");
                try {
                    Thread.sleep(1000);
                } catch (InterruptedException e) {
                    throw new RuntimeException(e);
                }

                synchronized(locker2) {
                    System.out.println("三叶_拿起第二把锁");
                    try {
                        Thread.sleep(1000);
                    } catch (InterruptedException e) {
                        throw new RuntimeException(e);
                    }
                }
            }

        },"三叶");

        Thread t2 = new Thread(()->{
            synchronized(locker2){
                System.out.println("折纸_拿起第二把锁");
                try {
                    Thread.sleep(1000);
                } catch (InterruptedException e) {
                    throw new RuntimeException(e);
                }
                synchronized(locker1) {
                    System.out.println("折纸_拿起第一把锁");
                    try {
                        Thread.sleep(1000);
                    } catch (InterruptedException e) {
                        throw new RuntimeException(e);
                    }
                }
            }

        },"折纸");

        t1.start();
        t2.start();

        t1.join();
        t2.join();

        System.out.println("main 结束");
 
```
代码意思其实是这样的。

首先三叶和折纸同时运行。
```java
        t1.start();
        t2.start();
```

三叶先拿到第一把锁（locker1），休息一秒钟，再去尝试获取第二把锁（locker2）
折纸先拿到第二把锁（locker2），休息一秒钟，再去尝试获取第一把锁（locker1）

在三叶``sleep(1000)``的时候，折纸已经拿到了第二把锁(locker2)，当折纸尝试获取第一把锁（locker1），此时这把锁已经在三叶的手上了。

程序可不会主动将锁释放掉，锁也不能强行获取。
这样死锁就构成了。

我们也可以通过jconsole看看线程之间的状态。

## 1.3 思索要构成的条件
这种情况我们是要避免的，不然程序就卡死了。
我们来看看到底如何才会造成死锁这种状态呢？

### 1. 互斥条件
锁只能被一个线程所占有.
小美只能我有，或者只能你有，但绝不能你有我也有....
### 2. 占有且等待
一个线程拿到一把锁后还想在去获取另一把锁。
我在玩电脑，我还想看电视。你正在看电视。但你见我又想看电视，还不想给将电脑给你玩。

### 3. 不可抢占
资源不能被强制抢走，只能由持有它的线程主动释放。
大家都是君子，你不能在我女朋友还喜欢我的情况下，抢走我的女朋友。

### 4.  循环等待
线程 1 等线程 2 的资源，线程 2 等线程 3 的资源……线程 N 又等线程 1 的资源。
我喜欢小美，小美喜欢你，你喜欢小爱，小爱喜欢我...
我们的都在等自己喜欢的那个人主动....

## 1.4 如何避免死锁？
我们已经知道了造成死锁的根本条件，那么避免死锁也就没那么困难了。
### 避免嵌套使用锁
我们刚才的例子就是在锁A里面尝试获取锁B。
```java
synchroized(锁1){
	//逻辑...
	synchronized(锁2){
		//逻辑...
		}
	}
```
减少嵌套的使用在一定程度上可以避免死锁.看例子

### 固定获取锁定的顺序
这危险
```java
// 线程1
lock(A);
lock(B);

// 线程2
lock(B);
lock(A);
```
这安全
```java
// 所有线程都按 A → B 的顺序加锁
lock(A);
lock(B);

```
在实际的开发环境中难免会遇上必须要写成嵌套。
```java
public void transferAtoB() {
    synchronized (A) {
        synchronized (B) {
            
        }
    }
}

public void transferBtoA() {
    synchronized (B) {
        synchronized (A) {
       
        }
    }
}
```
这种环境下通过规定获取锁的顺序，来避免死锁是非常有效的方法。
```java
public void func1() {
    synchronized (A) {
        synchronized (B) {
            
        }
    }
}

public void func2() {
    synchronized (A) {
        synchronized (B) {
       
        }
    }
}
```
只要func1还没有释放锁A，那么func2就只能进行等待。
不会出现func2拿到锁B的情况。保证了func1能够获取锁B并释放掉AB。

# 2. 内存可见性
## 引入
我们先来看这段代码
```java

import java.util.Scanner;

public class demo3 {
    private static  boolean flag = false;
    public static void main(String[] args) throws InterruptedException {
        Thread t1 = new Thread(()->{
            System.out.println("t1 开始运行");
            while(!flag){
             	; 
            }
            System.out.println("flag 已经被修改");
        },"折纸");
        Thread t2 = new Thread(()->{
            Scanner in = new Scanner(System.in);
            flag = in.nextBoolean();
            System.out.println("t2 已完成修改");
        },"三叶");
        t1.start();
        t2.start();
        t1.join();
        t2.join();
        System.out.println("main 已经结束");
    }
}

```
一般来说我们要是将flag改成true.那么程序肯定会运行终止。
因为!flag 为flase，循环进去不了。
但实际情况是折纸线程会一直死循环运行下去。
t2的改动好似并没有牵扯到t1中的flag。但实际上flag确实被改动为true了。
程序仍然死循环，就是内存可见性的问题了。
## 解释

1. 每个线程都有自己的 工作内存（缓存）

2. 共享变量（如 flag）存放在 主内存

3. 线程对变量的读写 不一定直接访问主内存

一开始程序刚开始运行，t1将flag的数值读取到了自己的寄存器中。
JVM为了优化程序运行速度，导致t1再后续，只读取寄存器上的值。
虽然flag的数值在内存上已经被修改成了true，但t1读取不到内存上。
从寄存器里取值，比从内存上找快了几个数量级。

我们可以通过关键`` volatile``来解决这个问题。
不让程序偷懒，强制读取内存上的值。

``private static volatile boolean flag = false;``
给变量加上一个volatile关键字，就可以达到效果了。

# 总结
1. 死锁的本质  
多线程竞争多个资源时，如果线程之间形成“互相等待、谁也不让谁”的局面，就会导致程序永久卡住，这就是死锁。

2. 死锁产生的四个必要条件  
互斥、占有且等待、不可抢占、循环等待 —— 四者同时满足，死锁就可能发生。

3. 避免死锁的核心思路  
尽量减少嵌套锁、统一加锁顺序、使用 tryLock 等非阻塞方式，都能有效降低死锁风险。

4. 内存可见性问题的根源  
线程有自己的工作内存，变量更新不一定立刻同步到主内存，导致其他线程“看不到最新值”。

5. volatile 的作用  
volatile 能保证变量的可见性（线程每次都从主内存读写），并禁止指令重排序，从而避免某些并发错误。
