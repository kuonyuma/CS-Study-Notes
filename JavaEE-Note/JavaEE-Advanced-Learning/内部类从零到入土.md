# 1. 内部类
内部类分为三种
1. 实例内部类
2. 静态内部类
3. 匿名内部类
---
## 1.1 实例内部类
简单来说，它被定义在外部类的成员位置，且没有 `static` 修饰。
这意味着它在逻辑地位上等同于外部类的一个**实例成员变量**或**实例方法**。
你无法在没有外部类对象的情况下单独创建一个内部类对象。
在内存层面，每一个实例内部类的对象都会隐式地持有一个指向外部类对象的引用。
```java
public class Outer {
    private String secret = "这是外部类的私有秘密";
    private static String staticSecret = "这是外部类的静态秘密";

    // 实例内部类：定义在成员位置，没有static
    class Inner {
        public void display() {
            // 直接访问外部类的私有成员和静态成员
            System.out.println("内部类访问: " + secret);
            System.out.println("内部类访问: " + staticSecret);
        }
    }
}
```
这种设计使得内部类可以像在自己家里一样，直接访问外部类所有的成员，包括那些被 `private` 严密保护的属性和方法。


在处理访问逻辑时，实例内部类遵循一套 **“就近原则”与“路径指定”**.
当内部类方法中出现变量名冲突时，编译器会默认优先指向内部类自己的变量。
如果你此时想要访问外部类的同名成员，就需要用到特殊的语法：`外部类名.this.变量名`。
```java
public class Outer {
    int x = 10;
    class Inner {
        int x = 20;
        public void printX() {
            int x = 30;
            System.out.println("局部变量: " + x);          // 结果：30
            System.out.println("内部类成员变量: " + this.x); // 结果：20
            // 关键点：通过 外部类名.this 访问外部类成员
            System.out.println("外部类成员变量: " + Outer.this.x); // 结果：10
        }
    }
}
```
这里的 `外部类名.this` 实际上就是内部类持有的那个隐式引用，它精准地指向了当前内部类对象所属的那个特定外部类实例。
注意：在 JDK 15 及之前的版本中，你不能在实例内部类中定义非 `final` 的静态成员。这是因为静态成员属于类级别，而实例内部类本身是实例级别的，这种逻辑上的冲突会导致编译器报错。但是，从 JDK 16 开始，Java 放宽了这个限制，实例内部类中也可以直接定义静态成员了。（如果你的环境是 JDK 8，请务必注意这个老规则）
```java
public class Outer {
    class Inner {
        // ❌ 编译报错（JDK 15 及之前）：Inner class cannot have static declarations (注：JDK 16起不再报错)
        // public static int count = 1; 

        // ✅ 但可以定义常量（编译时确定的值）
        public static final int CONSTANT = 100; 
    }
}
```

如果你想在外部类的静态方法或者其他无关类中创建内部类对象，你必须先亲手“造”出一个外部类对象，然后通过这个对象作为跳板来调用 `new` 操作符，语法形如 `Outter.Inner inner = outterInstance.new Inner();`。
```java
public class Test {
    public static void main(String[] args) {
        // 方式1：拆解动作
        Outer out = new Outer();
        Outer.Inner inner1 = out.new Inner();
        // 方式2：一气呵成（链式调用）
        Outer.Inner inner2 = new Outer().new Inner();   
        inner1.display();
    }
}
```

关于这个拆解写法我们可以补充一些东西。
这是你写的 
```java
Outer out = new Outer();
Outer.Inner inner1 = out.new Inner();
```
实际上可能是..
```java
Outer out = new Outer();
Outer.Inner inner1 = new Inner(out)
```
传递了一个参数，这个参数是外部类引用。
而且它还会加一个字段，并改你inner的构造方法
```java
// 1. 编译器给 Inner 类偷偷加了一个成员变量
class Inner {
    final Outer this$0; // 这里的 this$0 就是指向外部类的“锚点”
    // 2. 编译器偷偷把你的无参构造方法改成了带参构造方法
    public Inner(Outer outer) {
        this.this$0 = outer; // 把外面的“爸爸”对象存起来
    }
}
```
表面上你是这样访问外部类字段的。
```java
public class Outer {
    private String name = "张三";

    class Inner {
        public void showName() {
            // 表面上：直接写 name
            // 实际上：编译器将其转换为 Outer.this.name
            System.out.println("外部类的名字是：" + name);
        }
    }
}
```
实际上是..
```java
// 编译器眼中类似这样的逻辑（伪代码）
class Inner {
    final Outer this$0; // 隐藏的引用

    public Inner(Outer outer) {
        this.this$0 = outer; // 初始化时绑定外部类对象
    }

    public void showName() {
        // 通过隐藏引用访问外部类成员
        System.out.println("名字是：" + this$0.name); 
    }
}
```
这就引申出另一个问题：内部类含有一个隐藏的外部类引用。**如果这个内部类对象的存活时间比外部类还要长（比如内部类被丢进了一个长时间运行的线程里），那么它就会死死拽住外部类对象不放。这就导致就算外部类已经没用了，垃圾回收器（GC）也无法回收它，从而造成内存泄漏。**是的，Java也有内存泄漏！

##  1.2 静态内部类
静态内部类就是在上述实例内部类的基础上``static``关键字修饰的内部类。
静态内部类不能访问外部类的非静态字段与方法。
```java
package demo2;

public class Father {

    private static String schoolName = "清华大学";
    private String name;

    public Father(String name) {
        this.name = name;
    }

    public static void print(){
        System.out.println("来自父类的print() " + schoolName);
    }

    // 静态内部类 (Static Inner Class)
    // 1. 静态内部类可以直接访问外部类的静态成员（包括私有的）
    // 2. 静态内部类不可以直接访问外部类的非静态成员
    // 3. 静态内部类的实例化不需要依赖外部类对象
    public static class StaticInner {
        private String innerName;

        public StaticInner(String innerName) {
            this.innerName = innerName;
        }
        public StaticInner(){

        }

        public void show() {
            System.out.println("我是静态内部类：" + innerName);
            // 直接访问外部类的静态属性
            System.out.println("可以访问外部类的静态属性：" + schoolName);
            
            // 注意：这里无法直接访问外部类的实例属性 name，除非通过外部类实例
            // System.out.println(name); // 编译报错

            //方式1：
            print();
            //方式2：
            Father.print();
        }
    }

    public static void main(String[] args) {
        // 静态内部类的创建方式：外部类名.内部类名 变量名 = new 外部类名.内部类名();
        Father.StaticInner inner = new Father.StaticInner("三叶");
        inner.show();
    }
}
```
静态内部类一般用于什么呢？这里我们要注意到一个小细节。
静态内部类可以访问外部类的构造器，构造器是否私有并无关系。
我们想象这样的一个场景，假设我们要建造一台电脑出来，如果不使用静态内部类会发生什么？
```java
public class BuilderPattern {
    
    private String cpu;
    private String ram;
    private String storage;
    private String gpu;

    public BuilderPattern(......) {
        this.cpu = builder.cpu;
        this.ram = builder.ram;
        this.storage = builder.storage;
        this.gpu = builder.gpu;
    }
```
如果不使用静态内部类（Builder模式），当一个电脑类有十几个配置参数时，你要么得写一个包含十几个参数的巨型构造方法（极易传错顺序），要么得先 new 一个空电脑，然后写一堆 set 方法（这导致电脑在 set 完之前处于“半成品”状态，且无法做成不可变对象）。而 Builder 模式完美解决了这个问题。
如果加上静态内部类....
```java
package demo2;

/**
 * 教学案例：使用静态内部类实现建造者模式 (Builder Pattern)
 */
public class BuilderPattern {
    
    private String cpu;
    private String ram;
    private String storage;
    private String gpu;

    // 1. 私有化构造器，确保外部只能通过 Builder 来创建对象
    private BuilderPattern(Builder builder) {
        this.cpu = builder.cpu;
        this.ram = builder.ram;
        this.storage = builder.storage;
        this.gpu = builder.gpu;
    }

    @Override
    public String toString() {
        return "电脑配置清单: {" +
                "CPU='" + cpu + '\'' +
                ", 内存='" + ram + '\'' +
                ", 硬盘='" + storage + '\'' +
                ", 显卡='" + gpu + '\'' +
                '}';
    }

    /**
     * 2. 静态内部类 Builder
     * 场景：Builder 只需要负责收集参数，不需要访问外部类的实例成员，
     * 且必须独立于外部类实例存在（用于构建外部类实例），因此定义为 static。
     */
    public static class Builder {
        private String cpu;
        private String ram;
        private String storage;
        private String gpu;

        // 设置参数的方法，通过 return this 实现链式编程
        public Builder setCpu(String cpu) {
            this.cpu = cpu;
            return this;
        }

        public Builder setRam(String ram) {
            this.ram = ram;
            return this;
        }

        public Builder setStorage(String storage) {
            this.storage = storage;
            return this;
        }

        public Builder setGpu(String gpu) {
            this.gpu = gpu;
            return this;
        }

        // 最终调用 build 方法返回外部类对象
        public BuilderPattern build() {
            return new BuilderPattern(this);
        }
    }

    public static void main(String[] args) {
        // 演示：静态内部类的经典应用场景 - 链式构建复杂对象
        BuilderPattern myComputer = new BuilderPattern.Builder()
                .setCpu("Intel i9-13900K")
                .setRam("64GB DDR5")
                .setStorage("2TB NVMe SSD")
                .setGpu("RTX 4090")
                .build();

        System.out.println("--- 建造者模式演示 ---");
        System.out.println(myComputer);
    }
}

```
这样我们没有先new一个BuilderPattern对象就把电脑造出来了。
这里还有一个思想叫做链式编程。
看这一段代码
```java
 public Builder setCpu(String cpu) {
            this.cpu = cpu;
            return this;
        }
```
当执行 setCpu("Intel") 时，它完成了赋值，并把当前的 Builder 实例又丢回给了你。
当你写下这一串代码时，计算机会这样理解：	
new BuilderPattern.Builder()：先造一个空的“建造器”对象。
.setCpu("i9")：在这个建造器上贴个 CPU 标签，返回这个建造器。
.setRam("64G")：在同一个建造器上贴个内存标签，返回这个建造器。
.build()：最后告诉建造器，“贴好了，按照这些标签给我造个电脑吧！”

如果不这样写的话...
```java
Builder b = new Builder();
b.setCpu("i9");
b.setRam("64G");
b.setStorage("2T");
BuilderPattern computer = b.build();
```
这十分的繁琐。

## 1.3 匿名内部类
匿名内部类就是没有名字的内部类，我们在使用时才开始构建这个类。
因为有些内部类，会出现从始至终都只使用过一次，这种内部类，我们不必大费周章写一大串``..... ..... = new ....``这样的语法。

----
我们来看看这个类。
```java
public interface Runnable {
    public abstract void run();
}
```
Runnable 是 java.lang 包下的一个接口。这个接口里面只有一个抽象方法叫做run。
我们可以看见，这个方法里面什么都没有，仅仅只是一个类似声明的东西。
那么我们在使用的时候必然要自己重新实现这个方法。

----

重写的语法其实很简单。
```java
 Runnable runnable = new Runnable() {
            @Override
            public void run() {
                System.out.println("你好");
            }
        };
```

我们首先看不使用匿名内部类的写法
```java
public class MyTask implements Runnable {
    @Override
    public void run() {
        System.out.println("你好，我是独立的外部类");
    }
}
```
这种写法就是重新创建一个.java文件写一个MyTask类。
如何使用呢？
```java
Runnable runnable = new MyTask();
runnable.run();
```
可以看见我们上述使用匿名内部类的写法要简单很多。不需要重新创建一个文件，
在new这一步就直接到位了，简便许多。

---

我们知道内部类可以访问外部类的字段与方法。但匿名内部类在访问外部类方法里面的变量时，要注意到这个变量必须是final修饰或者等效于final。

我们来看这样的一个案例
```java
package demo3;

public class test2 {

    public static void main(String[] args) {
        Father father = new Father();
        father.test(20);

    }
}
class Father{

    public  void test(int age) {
        Runnable r = new Runnable() {
            @Override
            public void run() {
                System.out.println(age);
            }
        };
        r.run();
        System.out.println(age);
    }
}
```
实际上可能是...
```java
// 编译器生成的类
class Test$1 implements Runnable {
    private int val$age; // 偷偷创建的副本变量

    // 构造函数传参，把外部变量拷贝进来
    public Test$1(int age) {
        this.val$age = age;
    }

    @Override
    public void run() {
        System.out.println(this.val$age); // 实际上用的是自己的副本
    }
}
```
为什么要规定final？
核心原因是：
局部变量：存储在栈（Stack）上。方法执行结束，变量立即销毁。

匿名内部类对象：存储在堆（Heap）上。只要还有引用指向它，它就一直存在。

这就意味着，方法执行完了，局部变量都没了，但内部类对象可能还活着。为了防止内部类去访问一个已经“死掉”的变量，编译器非常聪明，它干脆偷偷把这个变量复制了一份放进了内部类里面（就像上面代码里的 `val$age`）。
既然是复制的，那万一你在外面把原来的 `age` 改了，或者在内部类里把复制的 `age` 改了，两边的数据不就不一致了吗？为了避免这种“数据不同步”的尴尬，Java 直接规定：这个变量必须是 `final` 的（或者实际上不发生改变），大家都不许改，这就没问题了。

---
匿名内部类在继承和实现上非常“专一”且吝啬：

不能多实现：只能实现一个接口。

不能既继承又实现：要么继承一个父类，要么实现一个接口，不能同时进行。

不能继承抽象类又实现接口。

# 总结

1. **实例内部类**：就像外部类的“亲儿子”，底层自带 外部类.this 隐式引用，能直接访问外部类所有成员，但如果不注意生命周期，容易引发内存泄漏。（注：JDK 16起已经允许定义静态成员啦）。
    
2. **静态内部类**：加了 static 的内部类，更像是一个寄宿在别人家里的独立类。它不需要依赖外部类对象就能创建，并且只能访问外部类的静态成员。日常开发中最常用来实现 **Builder（建造者）模式**。
    
3. **匿名内部类**：没有名字、用完即毁的“一次性类”。为了保证栈上的局部变量和堆上的对象数据一致，它要求访问的外部局部变量必须是 final 或等效于 final。它在继承和实现上非常专一，只能“二选一”。
