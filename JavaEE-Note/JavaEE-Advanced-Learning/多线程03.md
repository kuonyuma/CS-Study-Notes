# 1 线程等待

我们知道 sleep() 方法可以让一个线程暂时停止运行，比如 sleep(5000); 就是暂停运行 5 秒钟 (毫秒单位)。
但我们知道这只是让线程休眠一段时间而已，并不能有效地安排多个线程谁先结束，谁后结束。
请看这段代码：
```java
 Thread t = new Thread(() -> {
            System.out.println("快递开始派送");

            for(int i = 0; i < 3;i++) {
                System.out.println("派送中...");
                try {	
                	// 模拟任务耗时
                    Thread.sleep(1000); 
                } catch (InterruptedException e) {
                }
            }
            System.out.println("快递到了");
        });

        t.start();

        // 我猜快递 3 秒后能到，那我就睡 3 秒
        Thread.sleep(3000);

        System.out.println("取快递");
       
```

下面是可能的输出结果：
![在这里插入图片描述](https://i-blog.csdnimg.cn/direct/f4832cb1a7e74186a8d62d7f4b49023d.png)
由代码可以明白，我们 Thread 中的运行时间有 3 秒钟。所以我们在 main 线程中设置了一个 sleep(3000)。但是结果却是先取快递，但此时快递并没有送到。这并不合理。
因为线程之间是抢占式执行的，sleep 也不是准确的运行时间。

此时我们引入新的方法 join()；
以下是代码：
```java
Thread t = new Thread(() -> {
            System.out.println("快递开始派送");
            for(int i = 0; i < 3;i++) {
                System.out.println("派送中...");
                try {
                    Thread.sleep(1000); // 模拟任务耗时
                } catch (InterruptedException e) {
                }
            }
            System.out.println("快递到了");
        });
        t.start();
        //join
        t.join();
        // 我猜快递 3 秒后能到，那我就睡 3 秒
        //Thread.sleep(3000);
        System.out.println("取快递");
```
以下是运行结果：
![在这里插入图片描述](https://i-blog.csdnimg.cn/direct/e0777e579f4d44588c517773c3a0a0ec.png)
现在的运行结果就非常合理了，先等到快递送达，再去取快递。
现在我们要来讨论一下这个方法。


t.join(); 这行代码的含义非常明确：等待线程 $t$ 运行结束。
当 main 线程自上而下运行到这里时，如果 $t$ 还在忙碌，main 就会原地“挂起”进入阻塞状态，直到 $t$ 完成任务。
就像我们要等 $t$ 发出“快递送达”的信号，main 才能执行接下来的“取快递”。但……这种“死等”的方式真的科学吗？
仔细一想，其实并不合理。join() 的默认语义是：只要你不结束，我就绝不往后走。
有一天....
周日你想约暗恋的女神去逛街。女神发来信息：“周日 8 点，在我楼下碰头吧。”你欣喜若狂。
这天很快就到了。清晨 6 点你就等在了楼下。时间一秒一秒流逝，8 点到了，女神却没有出现。你心想：“女生嘛，出门总要打扮得漂亮点，再等等吧。”
10 点，你开始焦躁，在楼下来回踱步，频繁地解锁手机又熄屏。此时寒风凛冽，你手指不可屈伸。
11 点 30 分，女神终于下楼了。女神看了看你，似乎并没有因为你的提前到达和漫长等待而感动。
这合理吗？显然不啊。
 万一女神临时起意不来了，难道你要在那儿等上一整天，甚至一辈子吗？
 在编程中也是一样，我们不能把程序的命门完全交给另一个线程。
 所以，我们需要更科学地使用 join——给等待加一个期限。
 join(3000); 的含义是：我最多等你 3 秒。如果 3 秒到了你还没结束，那对不起，我不等了，我要继续执行我自己的逻辑。





# 2 线程状态
一般来讲我们讨论的状态在大体上分为两种：就绪与阻塞。
我们可以用 getState() 这个方法获取线程的状态。

今天我们细分一下，研究这几种状态：
1. NEW
2. TERMINATED
3. RUNNABLE
4. TIMED_WAITING
5. WAITING
6. BLOCKED
## 2.1 NEW
所谓的 NEW，也就是创建好了一个线程，但是线程还没开始运行，没有 .start()；
```java
 Thread t1 = new Thread(()->{
            System.out.println("我是折纸");
            for(int i = 0; i< 3;i++){
                try{
                    System.out.println("折纸送快递中...");
                    Thread.sleep(3000);
                } catch (InterruptedException e) {
                    throw new RuntimeException(e);
                }
            }
            System.out.println("你的快递已到达");
        },"折纸");

        System.out.println(t1.getState());
        t1.start();
        t1.join();
        System.out.println("main线程 结束");
```

## 2.4 TIMED_WAITING
这个表示阻塞状态。
这个状态是：
1. 有时间限制的阻塞，比如 join(1000);。
2. 此时 CPU 没有运行这个线程。

```java
 Thread t1 = new Thread(()->{
            System.out.println("我是折纸");
            for(int i = 0; i< 30000;i++){
                try{
                    System.out.println("折纸送快递中..." + i);
                    Thread.sleep(1000);
                } catch (InterruptedException e) {
                    throw new RuntimeException(e);
                }
            }
            System.out.println("你的快递已到达");
        },"折纸");
        t1.start();
        t1.join();
        System.out.println("main线程 结束");

```

# 总结

根据您的文章内容，总结如下：

1. **掌握 `join()` 方法的必要性**：由于线程是抢占式执行的，仅靠 `sleep()` 无法精确控制线程执行的先后顺序，而 `join()` 可以让一个线程明确等待另一个线程任务完成后再继续执行。
2. **科学等待的重要性**：无参的 `join()` 是“死等”，在实际开发中存在风险；建议使用带参数的 `join(millis)` 来设定超时时间，提高程序的健壮性。
3. **线程状态的生命周期**：线程从创建到销毁会经历不同的状态，通过 `getState()` 方法可以实时获取并分析线程当前的运行情况。
4. **理解常见线程状态**：
   - **NEW**：已创建但未启动。
   - **RUNNABLE**：就绪状态，随时待命或正在运行。
   - **TERMINATED**：任务执行完毕，但对象尚未销毁。
5. **区分两种阻塞状态**：`WAITING` 是无限期的死等，而 `TIMED_WAITING` 是带有时间期限的等待（如 `sleep` 或带参数的 `join`）。
6. **后续进阶方向**：针对锁竞争导致的 `BLOCKED` 状态，是后续深入学习多线程并发与同步机制的核心内容。
